# Debugging with HIP[\#](#debugging-with-hip "Link to this heading"){.headerlink}

HIP debugging tools include [ltrace](https://ltrace.org/){.reference .external} and [[ROCgdb]{.xref .std .std-doc}](https://rocm.docs.amd.com/projects/ROCgdb/en/latest/index.html "(in ROCgdb Documentation v16.3)"){.reference .external}. External tools are available and can be found online. For example, if you're using Windows, you can use Microsoft Visual Studio and WinGDB.

You can trace and debug your code using the following tools and techniques.

::::
{#tracing .section}
## Tracing[\#](#tracing "Link to this heading"){.headerlink}

You can use tracing to quickly observe the flow of an application before reviewing the detailed information provided by a command-line debugger. Tracing can be used to identify issues ranging from accidental API calls to calls made on a critical path.

ltrace is a standard Linux tool that provides a message to [`stderr`{.docutils .literal .notranslate}]{.pre} on every dynamic library call. You can use ltrace to visualize the runtime behavior of the entire ROCm software stack.

Here's a simple command-line example that uses ltrace to trace HIP APIs and output:

:
{.highlight-console .notranslate}

highlight
    $ ltrace -C -e "hip*" ./hipGetChanDesc

## Debugging[\#](#debugging "Link to this heading"){.headerlink}

You can use ROCgdb for debugging and profiling.

ROCgdb is the ROCm source-level debugger for Linux and is based on GNU Project debugger (GDB). the GNU source-level debugger, equivalent of CUDA-GDB, can be used with debugger frontends, such as Eclipse, Visual Studio Code, or GDB dashboard. For details, see ([ROCm/ROCgdb](https://github.com/ROCm/ROCgdb){.github .reference .external}).

Below is a sample how to use ROCgdb run and debug HIP application, ROCgdb is installed with ROCM package in the folder /opt/rocm/bin.

## Useful environment variables[\#](#useful-environment-variables "Link to this heading"){.headerlink}

HIP provides environment variables that allow HIP, hip-clang, or HSA drivers to prevent certain features and optimizations. These are not intended for production, but can be useful to diagnose synchronization problems in the application (or driver).

Some of the more widely used environment variables are described in this section. These are supported on the Linux ROCm path and Windows.

{#kernel-enqueue-serialization .section}
### Kernel enqueue serialization[\#](#kernel-enqueue-serialization "Link to this heading"){.headerlink}

You can control kernel command serialization from the host:

[`AMD_SERIALIZE_KERNEL`{.docutils .literal .notranslate}]{.pre}, for serializing kernel enqueue

{#making-device-visible .section}
### Making device visible[\#](#making-device-visible "Link to this heading"){.headerlink}

For systems with multiple devices, you can choose to make only certain device(s) visible to HIP using [`HIP_VISIBLE_DEVICES`{.docutils .literal .notranslate}]{.pre} (or [`CUDA_VISIBLE_DEVICES`{.docutils .literal .notranslate}]{.pre} on an NVIDIA platform). Once enabled, HIP can only view devices that have indices present in the sequence. For example:

:
{.highlight-console .notranslate}

highlight
    $ HIP_VISIBLE_DEVICES=0,1

:

Or in the application:

:
{.highlight-cpp .notranslate}

highlight
    if (totalDeviceNum > 2) {
    setenv("HIP_VISIBLE_DEVICES", "0,1,2", 1);
    assert(getDeviceNumber(false) == 3);
    ... ...
    }

:

{#hsa-related-environment-variables-linux .section}
### HSA-related environment variables (Linux)[\#](#hsa-related-environment-variables-linux "Link to this heading"){.headerlink}

HSA provides environment variables that help analyze issues in drivers or hardware.

- To isolate issues with hardware copy engines, you can use [`HSA_ENABLE_SDMA`{.docutils .literal .notranslate}]{.pre}.

  [`HSA_ENABLE_SDMA=0`{.docutils .literal .notranslate}]{.pre} causes host-to-device and device-to-host copies to use compute shader blit kernels, rather than the dedicated DMA copy engines. Compute shader copies have low latency (typically \< 5 us) and can achieve approximately 80% of the bandwidth of the DMA copy engine.

- To diagnose interrupt storm issues in the driver, you can use [`HSA_ENABLE_INTERRUPT`{.docutils .literal .notranslate}]{.pre}.

  [`HSA_ENABLE_INTERRUPT=0`{.docutils .literal .notranslate}]{.pre} causes completion signals to be detected with memory-based polling, rather than interrupts.

## General debugging tips[\#](#general-debugging-tips "Link to this heading"){.headerlink}

- [`gdb`{.docutils .literal .notranslate}]{.pre}` `{.docutils .literal .notranslate}[`--args`{.docutils .literal .notranslate}]{.pre} can be used to pass the executable and arguments to [`gdb`{.docutils .literal .notranslate}]{.pre}.

- You can set environment variables ([`set`{.docutils .literal .notranslate}]{.pre}` `{.docutils .literal .notranslate}[`env`{.docutils .literal .notranslate}]{.pre}) from within GDB on Linux:

  :
{.highlight-bash .notranslate}
  
highlight
      (gdb) set env AMD_SERIALIZE_KERNEL 3
  
  :

  
{.admonition .note}
  Note

  This [`gdb`{.docutils .literal .notranslate}]{.pre} command does not use an equal (=) sign.
  

- The GDB backtrace shows a path in the runtime. This is because a fault is caught by the runtime, but it is generated by an asynchronous command running on the GPU.

- To determine the true location of a fault, you can force the kernels to run synchronously by setting the environment variables [`AMD_SERIALIZE_KERNEL=3`{.docutils .literal .notranslate}]{.pre} and [`AMD_SERIALIZE_COPY=3`{.docutils .literal .notranslate}]{.pre}. This forces HIP runtime to wait for the kernel to finish running before returning. If the fault occurs when a kernel is running, you can see the code that launched the kernel inside the backtrace. The thread that's causing the issue is typically the one inside [`libhsa-runtime64.so`{.docutils .literal .notranslate}]{.pre}.

- VM faults inside kernels can be caused by:

  - Incorrect code (e.g., a for loop that extends past array boundaries)

  - Memory issues, such as invalid kernel arguments (null pointers, unregistered host pointers, bad pointers)

  - Synchronization issues

  - Compiler issues (incorrect code generation from the compiler)

  - Runtime issues
